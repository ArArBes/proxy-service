# Proxy Service

Сервис для предоставления доступа к прокси-серверам по одноразовому ключу активации.  
Пользователь регистрируется на сайте, получает ключ на email, затем использует его в десктопном приложении 
для подключения к свободной вм.

## Технологии

- **Backend**: Django 5.2, Django REST Framework, Django Channels
- **База данных**: PostgreSQL
- **Асинхронные задачи**: Celery + Redis
- **Веб-сокеты**: Daphne + Channels
- **Веб-интерфейс**: Vue 3 + Vuetify
- **Десктопное приложение**: Tkinter, `requests`, `websocket-client`
- **Контейнеризация**: Docker, docker-compose
- **Документация API**: drf-spectacular (Swagger)
- **Тесты**: pytest, pytest-django, pytest-asyncio
- **SMTP для разработки**: Mailpit

## Файл окружения

В корне проекта уже есть файл `.env`, содержащий все необходимые переменные (настройки базы данных, Redis, Celery, Django secret key и т.д.). 

## Запуск проекта

### Предварительные требования

- Docker и Docker Compose
- Node.js (для разработки фронтенда)
- Python 3.11+ (для десктопного приложения)

### Способ 1. Запуск через Makefile

1. **Клонируйте репозиторий**
   ```bash
   git clone <url>
   cd proxy-service

2. Запуск бэкенда (Django + PostgreSQL + Redis + Celery + Nginx + Mailpit)
```bash
make up
```

**Что происходит:**
- Команда поднимет все контейнеры в фоновом режиме
- Скрипт `entrypoint.sh` автоматически применит миграции и создаст три тестовые виртуальные машины 

### 3. Запуск фронтенда (Vue)
**В отдельном терминале**:
```bash
make front
```

**После этого фронтенд будет доступен по адресу:** http://localhost:5173

### 4. Запуск десктопного приложения (Tkinter)
**В третьем терминале** выполните:
```bash
make desktop
```

**Что происходит:**
- Приложение создаст виртуальное окружение
- Установит зависимости
- Запустится автоматически

## Тестовые данные

При запуске бэкенда **автоматически** (через скрипт `entrypoint.sh` и management command `create_test_vms`) создаются **три виртуальные машины** для тестирования:

| Имя      | Тип     | Хост          | Порт  |
|----------|---------|---------------|-------|
| proxy-1  | socks5  | 192.168.1.10  | 1080  |
| proxy-2  | http    | 192.168.1.11  | 8080  |
| proxy-3  | https   | 192.168.1.12  | 443   |

## Тестирование

### Запуск тестов

**В контейнере backend:**
```bash
docker-compose exec backend pytest -v
```

**Или через Makefile:**
```bash
make test
```

### Что проверяют тесты

#### Приложение `users`
- **Модель `User` и `UserManager`**: создание пользователя, генерация ключа, валидация ключа
- **Сериализаторы**: регистрация, логин, профиль, смена пароля, активационный ключ
- **Представления**: регистрация, логин, профиль, смена пароля, обновление ключа
- **Сервисы**: создание JWT-токенов, удаление/обновление ключа

#### Приложение `virtual_machines`
- **Модель `VirtualMachine`**: строковое представление, сериализация данных
- **Сервис `connect_user_to_vm`**: поиск свободной ВМ, назначение пользователю
- **Представление `ActivateKeyAndGetVmView`**: активация ключа, выдача параметров прокси
- **WebSocket consumer `VMConnect`**: установка соединения, обработка отключения, освобождение ВМ

## Архитектура взаимодействия

1. Пользователь регистрируется через Vue-интерфейс, запрос уходит в Django через nginx.
2. После регистрации Celery асинхронно отправляет письмо с ключом активации через Mailpit.
3. Пользователь вставляет ключ в десктопное приложение (Tkinter), отправляет запрос на `/api/activate-key/`.
4. Бэкенд проверяет ключ, удаляет ключ, назначает свободную виртуальную машину и возвращает её параметры.
5. Сразу после этого клиент открывает WebSocket-соединение (Django Channels + Daphne + Redis) и получает статусы:
   - `connecting`
   - `connected` 
   - `disconnected`
6. При отключении виртуальная машина освобождается (`current_user_id = None`).
